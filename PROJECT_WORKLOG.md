## Границы проекта

В git и в будущий GitHub-репозиторий можно выносить только папку `crystal_stone`.
Это нормальная схема.

Если инициализировать git именно внутри `crystal_stone`, то:

- файлы из корневой папки `voximplant` в этот репозиторий не попадут;
- для других папок позже можно завести отдельные независимые репозитории;
- история `Crystal Stone` будет изолирована от остальных сценариев.

Рекомендуемая стратегия:

1. Отдельный repo root: `voximplant/crystal_stone`
2. Отдельный `.gitignore` только для `crystal_stone`
3. Не хранить в git:
   - `backend/.env`
   - `backend/venv/`
   - `backend/voximplant.db`
   - локальные записи звонков
   - логи
   - временные бэкапы

### Что найдено при аудите

1. `crystalstone_server_edition.js` сейчас синтаксически сломан и не проходит `node --check`.
2. Серверная миграция недоведена до конца:
   - backend ожидает lifecycle-события `call_started`, `call_finished`, `recording_ready`;
   - текущий server-edition сценарий их не отправляет;
   - в backend из сценария реально уходят только `report` и `google_sheets`.
3. Рабочие старые сценарии все еще содержат прямую интеграцию с Telegram / Google Sheets.
4. Backend сейчас небезопасен для продакшена:
   - нет аутентификации webhook-ов;
   - публично доступны записи;
   - публично доступен список звонков.
5. База данных слишком простая для целевой истории звонков:
   - нет полноценного хранения итогового payload;
   - нет отдельного хранения диалога, токенов, стоимости, статусов доставки.
6. Надежной очереди доставки нет:
   - отправка в Google Sheets идет fire-and-forget;
   - скачивание записи идет fire-and-forget;
   - нет retry-state / outbox / dead-letter логики.

## Целевое состояние архитектуры

Цель:
Voximplant должен стать тонким клиентом, а вся прикладная логика должна уйти на backend.

Целевой поток:

1. Voximplant ведет звонок и получает итоговые данные.
2. Voximplant шлет только события и финальный structured payload на backend.
3. Backend:
   - сохраняет историю звонка в БД;
   - скачивает запись;
   - отправляет сообщения в Telegram;
   - синхронизирует Google Sheets;
   - управляет TTL и очисткой;
   - при необходимости отдает данные для CRM / админки / аналитики.
