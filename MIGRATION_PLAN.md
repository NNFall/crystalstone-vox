# План миграции логики Crystal Stone на собственный сервер

Цель: Перенести бизнес-логику из Voximplant-сценариев на внешний сервер (Backend API), чтобы упростить обслуживание, обеспечить сохранность записей и централизовать управление данными.

## 1. Архитектурные изменения
- **Voximplant:** Остается "тонким клиентом" (только обработка потока аудио и событий звонка).
- **Сервер (Backend):** 
    - Принимает события звонка от Voximplant (Webhook/HTTP).
    - Взаимодействует с Google Apps Script (интеграция с таблицами).
    - Управляет БД звонков (PostgreSQL/SQLite).
    - Загружает аудиозаписи звонков.
    - Выполняет функции Telegram-бота.

## 2. Этапы реализации

### Этап 1: Инфраструктура сервера
- Развертывание базового API-сервера (Node.js/Python).
- Настройка endpoints для приема событий от Voximplant (`call_started`, `call_finished`, `recording_ready`).
- Настройка БД для хранения метаданных звонков.

### Этап 2: Интеграция с сервисами
- Перенос логики отправки данных в Google Sheets с клиентской стороны (Voximplant) на сервер.
- Реализация логики загрузки записей (Voximplant -> Сервер).
- Настройка ротации записей (удаление старых файлов по истечении срока).

### Этап 3: Telegram-бот
- Перенос всей логики уведомлений и команд с Voximplant на сервер.
- Сервер теперь сам отправляет отчеты о завершенных звонках в ТГ.

### Этап 4: Рефакторинг Voximplant
- Очистка `crystalstone_2_5_flash.js` от кода логики и интеграций.
- Настройка сценария на отправку только минимального набора данных (ID звонка, события, ссылка на запись).

## 3. Стек технологий
- **Backend:** Python (FastAPI + aiogram для Telegram-бота).
- **База данных:** SQLite.
- **Хранилище:** Локальная файловая система.
